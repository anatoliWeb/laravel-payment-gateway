# Activity Logging

## Purpose

Activity logs provide an operational audit trail for high-signal billing actions.

They help admin/operator UI, support review, and portfolio demos explain what happened across payment creation, simulation, subscriptions, idempotency, webhooks, usage limits, wallet adjustments, invoices, and scheduler actions.

Activity logs are not a financial ledger. Payment, invoice, wallet transaction, payment transaction, and webhook delivery tables remain the source of financial and delivery truth.

## Non-Goals

- no raw card, CVV/CVC, PAN, token, password, provider secret, private key, or encrypted credential storage
- no raw idempotency key storage
- no full callback URL in activity metadata when callback host is enough
- no noisy catalog-browse logs for every plan view
- no replacement for payment transactions, invoice totals, or wallet ledger rows

## Action Naming

Billing activity actions use stable string names so admin UI filters can rely on them.

Canonical pattern:
- `billing.<domain>_<event>`
- `chat.feature_limit_exceeded` remains the existing chat-owned billing-limit action

Existing aliases are kept when already implemented and tested. For example, payment success/failure simulation uses `billing.payment_simulated_success` and `billing.payment_simulated_failure` instead of renaming existing records to `billing.payment_succeeded` / `billing.payment_failed`.

## Payment Activity Logs

Implemented:
- `billing.payment_created`
- `billing.payment_simulated_success`
- `billing.payment_simulated_failure`
- `billing.payment_risk_blocked`
- `billing.payment_suspicious_attempt`

Safe metadata includes payment id/uuid, status, amount, currency, payment source, company id, seller id, payer user id, provider, safe provider reference where already simulator-generated, and safe simulation metadata.

## Subscription Activity Logs

Implemented:
- `billing.subscription_created`
- `billing.subscription_activated`
- `billing.subscription_payment_failed`
- `billing.subscription_plan_upgrade_requested`
- `billing.subscription_plan_downgrade_requested`
- `billing.subscription_cancelled`
- `billing.subscription_expired`
- `billing.subscription_renewal_attempted`
- `billing.subscription_renewal_succeeded`
- `billing.subscription_renewal_failed`
- `billing.subscription_past_due`

Safe metadata includes subscription id/uuid, plan id, status, related payment id/uuid, and lifecycle source.

## Invoice Activity Logs

Implemented invoice service logs include invoice creation, issue, payment-pending, paid, failed, and void flows.

Safe metadata should include invoice id/uuid/number, status, totals, currency, payment id when present, and ownership scope.

## Wallet Activity Logs

Implemented wallet logs include:
- wallet top-up success
- auto top-up / auto charge consent changes
- manual wallet credit/debit adjustments

Wallet activity logs are audit markers only. Wallet transaction rows remain the ledger.

## Idempotency Activity Logs

Implemented:
- `billing.idempotency_replayed`
- `billing.idempotency_conflict`

Metadata stores safe context only: idempotency row id, hashed key, scope, status, related resource type/id, and response status.

Raw idempotency keys are never logged.

## Webhook Activity Logs

Implemented:
- `billing.webhook_delivery_created`
- `billing.webhook_dispatched`
- `billing.webhook_delivered`
- `billing.webhook_failed`
- `billing.webhook_permanently_failed`
- `billing.webhook_retry_requested`

Safe metadata includes delivery id/uuid, payment id, event type, status, attempts, callback host, company id, and seller id.

Full callback URLs, signatures, and webhook secrets are not activity metadata.

## Usage Limit Activity Logs

Implemented:
- `chat.feature_limit_exceeded`

Safe metadata includes attempted action, feature key, reason, used value, and limit value. The action is chat-owned but categorized as billing-related through metadata.

## Scheduler Activity Logs

Implemented scheduler actions cover expired payments, usage reset, webhook retry queueing, cleanup, subscription expiration, and subscription renewal decisions.

Scheduler logs should stay coarse-grained. They should describe batch outcomes and touched entity ids, not every low-value loop step.

## Metadata Safety

Never log:
- raw idempotency keys
- raw card number, PAN, CVV, CVC, security code
- provider secrets, private keys, encrypted credentials
- full unsafe request metadata
- full webhook signature secrets

Allowed:
- internal ids
- public UUIDs
- hashed idempotency key values
- amounts/currency/statuses
- callback host without path/query
- simulator provider references generated by this system

## Testing Strategy

`CriticalActivityLogsTest` verifies:
- critical payment, subscription, idempotency, webhook, and usage limit actions exist
- idempotency replay/conflict logs are written
- webhook dispatch/delivered/failed logs are written
- activity metadata does not contain raw idempotency keys or obvious payment/security secret values

Related regression tests continue to validate the owning business flows.
